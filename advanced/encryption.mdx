title: State Encryption
description: Secure your Terraform state with encryption at rest

OpenTofu supports client-side state encryption to protect sensitive data in your state files. The ORAS backend works seamlessly with OpenTofu's encryption features to ensure your state is encrypted before being stored in the registry.

## Overview

State encryption is configured in the `encryption` block within your Terraform configuration. The encryption happens client-side before the state is pushed to the OCI registry, ensuring your sensitive data is never transmitted or stored unencrypted.

<Warning>
Encryption requires OpenTofu or a Terraform distribution that supports state encryption. Standard Terraform OSS does not support the `encryption` block.
</Warning>

## Basic Encryption Setup

The most common encryption setup uses PBKDF2 for key derivation and AES-GCM for encryption:

```hcl
terraform {
  backend "oras" {
    repository = "ghcr.io/myorg/infra-state"
  }

  encryption {
    key_provider "pbkdf2" "main" {
      passphrase = var.state_passphrase
    }
    method "aes_gcm" "main" {
      key_provider = key_provider.pbkdf2.main
    }
    state { method = method.aes_gcm.main }
    plan  { method = method.aes_gcm.main }
  }
}

variable "state_passphrase" {
  type      = string
  sensitive = true
}
```

### Passphrase Management

Provide the passphrase via environment variable:

```bash
export TF_VAR_state_passphrase="your-secure-passphrase"
ghoten plan
```

<Note>
Never commit passphrases to version control. Use environment variables, secret management systems, or secure CI/CD variable storage.
</Note>

## Production Encryption Setup

For production environments, use a KMS-backed key provider instead of PBKDF2:

```hcl
terraform {
  backend "oras" {
    repository   = "ghcr.io/myorg/prod-state"
    compression  = "gzip"
    max_versions = 10
  }

  encryption {
    key_provider "aws_kms" "main" {
      kms_key_id = "arn:aws:kms:us-east-1:123456789012:key/12345678-1234-1234-1234-123456789012"
      region     = "us-east-1"
    }
    method "aes_gcm" "main" {
      key_provider = key_provider.aws_kms.main
    }
    state { method = method.aes_gcm.main }
    plan  { method = method.aes_gcm.main }
  }
}
```

## Encryption Components

### Key Providers

Key providers generate or retrieve the encryption keys:

- **`pbkdf2`** - Password-based key derivation (development/testing)
- **`aws_kms`** - AWS Key Management Service (production)
- **`azure_keyvault`** - Azure Key Vault (production)
- **`gcp_kms`** - Google Cloud KMS (production)

### Encryption Methods

- **`aes_gcm`** - AES-256 in GCM mode (recommended for all use cases)
- **`unencrypted`** - No encryption (for selective encryption scenarios)

### Encryption Targets

- **`state`** - Encrypts the state file
- **`plan`** - Encrypts plan files
- **`remote_state_data_sources`** - Encrypts remote state data source outputs

## Example: Multiple Environments

Use different encryption configurations for different environments:

```hcl
# production.tfvars
terraform {
  backend "oras" {
    repository = "ghcr.io/myorg/prod-state"
  }

  encryption {
    key_provider "aws_kms" "main" {
      kms_key_id = var.prod_kms_key_id
      region     = "us-east-1"
    }
    method "aes_gcm" "main" {
      key_provider = key_provider.aws_kms.main
    }
    state { method = method.aes_gcm.main }
  }
}
```


## Encryption with Compression

Encryption and compression can be used together. The backend compresses state before encryption:

```hcl
terraform {
  backend "oras" {
    repository  = "ghcr.io/myorg/infra-state"
    compression = "gzip"
  }

  encryption {
    key_provider "pbkdf2" "main" {
      passphrase = var.state_passphrase
    }
    method "aes_gcm" "main" {
      key_provider = key_provider.pbkdf2.main
    }
    state { method = method.aes_gcm.main }
  }
}
```

<Note>
Compression happens before encryption. Encrypted data does not compress well, so compression must be applied first.
</Note>

## Key Rotation

To rotate encryption keys:

1. Update your key provider configuration with the new key
2. Run `ghoten init` to reconfigure the backend
3. Run `ghoten apply` to re-encrypt the state with the new key

The state is automatically re-encrypted on the next write operation.

## Security Best Practices

<Warning>
**Critical Security Recommendations:**

- Never use PBKDF2 with passphrases in production—always use a KMS provider
- Ensure KMS keys have appropriate access policies and audit logging enabled
- Use different encryption keys for different environments
- Regularly rotate encryption keys according to your security policy
- Store passphrases and KMS key IDs in secure secret management systems
</Warning>

### Access Control

Combine encryption with registry authentication:

```bash
# Authenticate to registry
docker login ghcr.io

# Set encryption passphrase
export TF_VAR_state_passphrase="$(aws secretsmanager get-secret-value --secret-id state-passphrase --query SecretString --output text)"

# Run Terraform operations
ghoten apply
```

## Troubleshooting Encryption

### Error: Decryption failed

This typically indicates the wrong passphrase or key is being used:

```
Error: Failed to load state: decryption failed
```

**Solution**: Verify you're using the same passphrase or KMS key that was used to encrypt the state.

### Error: Encryption not supported

```
Error: Unsupported block type: "encryption"
```

**Solution**: Ensure you're using OpenTofu or a Terraform distribution that supports state encryption. Standard Terraform OSS does not support the `encryption` block.

### Error: KMS access denied

```
Error: Failed to encrypt state: AccessDeniedException
```

**Solution**: Verify your AWS credentials have `kms:Encrypt` and `kms:Decrypt` permissions for the specified KMS key.
