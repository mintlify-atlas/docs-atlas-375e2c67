Thank you for your interest in contributing to Ghoten! This guide provides guidelines and information to help you get started.

## Code of Conduct

This project adheres to a Code of Conduct. By participating, you are expected to uphold this code. Please report unacceptable behaviour to the maintainers.

## How can I contribute?

### Reporting bugs

<Steps>
  <Step title="Check existing issues">
    Search [existing issues](https://github.com/vmvarela/ghoten/issues) before creating a new one to avoid duplicates.
  </Step>
  
  <Step title="Use the Bug Report template">
    When creating a new issue, use the **Bug Report** issue template.
  </Step>
  
  <Step title="Include reproduction details">
    Provide:
    - Steps to reproduce the bug
    - Expected vs. actual behaviour
    - Relevant logs with debug output:
      ```bash
      TF_LOG=DEBUG ghoten plan
      ```
  </Step>
</Steps>

### Suggesting features

<Steps>
  <Step title="Open a Feature Request">
    Use the **Feature Request** issue template.
  </Step>
  
  <Step title="Describe the feature">
    Include:
    - The problem it solves
    - Your proposed solution
    - Alternative approaches you considered
  </Step>
</Steps>

### Submitting code

<Steps>
  <Step title="Fork the repository">
    Create your own fork of the Ghoten repository.
  </Step>
  
  <Step title="Create a feature branch">
    Branch from `master`:
    ```bash
    git checkout -b feature/my-change
    ```
  </Step>
  
  <Step title="Make your changes">
    Implement your changes and write tests for them.
  </Step>
  
  <Step title="Submit a pull request">
    Create a pull request targeting the `master` branch.
  </Step>
</Steps>

## Development setup

### Prerequisites

- **Go** - version specified in [go.mod](https://github.com/vmvarela/ghoten/blob/master/go.mod)
- **Make** - for build automation
- **Docker** (optional) - for integration tests and image builds

### Build from source

```bash
git clone https://github.com/vmvarela/ghoten.git
cd ghoten
make build
```

See the [Building from Source](/development/building) page for detailed build instructions.

### Run tests

```bash
# Full test suite
make test

# With coverage
make test-with-coverage

# ORAS backend only
go test ./internal/backend/remote-state/oras/...
```

See the [Testing](/development/testing) page for comprehensive testing documentation.

### Run linter

```bash
make golangci-lint
```

## Pull request process

<Steps>
  <Step title="Ensure tests pass">
    ```bash
    make test
    ```
  </Step>
  
  <Step title="Ensure code passes linting">
    ```bash
    make golangci-lint
    ```
  </Step>
  
  <Step title="Tidy dependencies (if changed)">
    If you modified dependencies:
    ```bash
    go mod tidy
    ```
  </Step>
  
  <Step title="Update documentation">
    If you changed user-facing behaviour, update the relevant documentation.
  </Step>
  
  <Step title="Fill out the PR template">
    Complete the pull request template with all required information.
  </Step>
  
  <Step title="Wait for review">
    A maintainer will review your PR. Address any requested changes promptly.
  </Step>
</Steps>

## Commit message guidelines

Use clear, descriptive commit messages. We prefer the [Conventional Commits](https://www.conventionalcommits.org/) format:

<CodeGroup>
```bash Feature addition
feat: add retry configuration for ORAS backend
```

```bash Bug fix
fix: correct lock TTL calculation
```

```bash Documentation
docs: update installation instructions
```

```bash Refactoring
refactor: extract state locking into separate package
```

```bash Tests
test: add integration tests for S3 backend
```
</CodeGroup>

## Coding standards

<AccordionGroup>
  <Accordion title="Code style">
    - Follow the existing code style and conventions in the project
    - Use `gofmt` / `goimports` for formatting
    - All exported types and functions must have doc comments
  </Accordion>
  
  <Accordion title="Testing">
    - Write unit tests for new functionality
    - Ensure tests are deterministic and can run in any order
    - Use table-driven tests where appropriate
  </Accordion>
  
  <Accordion title="Pull requests">
    - Keep changes focused - one logical change per PR
    - Avoid mixing refactoring with feature work
    - Break large changes into smaller, reviewable PRs
  </Accordion>
  
  <Accordion title="Documentation">
    - Update user-facing documentation for behavior changes
    - Add code comments for complex logic
    - Document exported APIs thoroughly
  </Accordion>
</AccordionGroup>

## Reporting security issues

<Warning>
  **Do not open a public issue for security vulnerabilities.**
</Warning>

Please see [SECURITY.md](https://github.com/vmvarela/ghoten/blob/master/SECURITY.md) for responsible disclosure instructions.
