## Overview

This guide gets you from zero to a working `ghoten plan` with an OCI-backed state in minutes. It focuses on the happy path and skips optional configuration.

<Note>
For full backend tuning (retry, rate limits, TLS options), see the [ORAS Backend Guide](/concepts/oras-backend) after completing this quickstart.
</Note>

## Prerequisites

- Git installed
- Docker installed (for registry authentication)
- A GitHub account with a personal access token (for GHCR)
- Go 1.21+ (if building from source)

## Installation

<Steps>
  <Step title="Clone the repository">
    Clone Ghoten from GitHub:
    
    ```bash
    git clone https://github.com/vmvarela/ghoten.git
    cd ghoten
    ```
  </Step>
  
  <Step title="Build the binary">
    Build `ghoten` locally using the provided Makefile:
    
    ```bash
    make build
    ```
    
    This compiles the binary to `./ghoten` in your current directory.
  </Step>
  
  <Step title="Verify installation">
    Check that the build succeeded:
    
    ```bash
    ./ghoten version
    ```
    
    You should see output similar to:
    
    ```
    OpenTofu v1.8.0
    on linux_amd64
    ```
  </Step>
</Steps>

<Accordion title="Using Docker instead">
For CI or reproducible local environments, you can use the published image:

```bash
docker pull ghcr.io/vmvarela/ghoten:<version>
```

Replace `<version>` with the specific release tag you want to use.
</Accordion>

## Configure the ORAS Backend

<Steps>
  <Step title="Create a basic configuration">
    In your infrastructure directory, create or update your `main.tf` with the minimal ORAS backend:
    
    ```hcl
    terraform {
      backend "oras" {
        repository = "ghcr.io/acme/infra-state"
      }
    }
    ```
    
    Replace `acme` with your GitHub username or organization.
    
    <Warning>
    The `repository` parameter is **required**. Everything else is optional tuning.
    </Warning>
  </Step>
  
  <Step title="Authenticate to the registry">
    Login to GitHub Container Registry using Docker:
    
    ```bash
    echo "$GITHUB_TOKEN" | docker login ghcr.io -u YOUR_GITHUB_USER --password-stdin
    ```
    
    Replace `YOUR_GITHUB_USER` with your GitHub username.
    
    <Note>
    Your `GITHUB_TOKEN` needs `read:packages` and `write:packages` scopes.
    </Note>
  </Step>
</Steps>

## Initialize and Run

<Steps>
  <Step title="Initialize the backend">
    Initialize your OpenTofu configuration:
    
    ```bash
    ./ghoten init
    ```
    
    You should see:
    
    ```
    Initializing the backend...
    
    Successfully configured the backend "oras"!
    ```
  </Step>
  
  <Step title="Run your first plan">
    Execute a plan to verify everything works:
    
    ```bash
    ./ghoten plan
    ```
    
    Your state is now stored in GHCR as an OCI artifact.
  </Step>
</Steps>

<Accordion title="Debugging issues">
If something fails, enable debug logging immediately instead of guessing:

```bash
TF_LOG=DEBUG ./ghoten plan
```

This provides detailed information about authentication, registry communication, and state operations.
</Accordion>

## Production Baseline

Once the minimal setup works, upgrade to this production baseline:

```hcl
terraform {
  backend "oras" {
    repository   = "ghcr.io/myorg/infra-state"
    compression  = "gzip"
    lock_ttl     = 300
    max_versions = 10
  }
}
```

**What each parameter does:**

- `compression = "gzip"` - Reduces state file size for faster transfers
- `lock_ttl = 300` - Auto-clears stale locks after 5 minutes (avoids manual cleanup after crashed jobs)
- `max_versions = 10` - Keeps 10 historical versions (maintains history without unbounded growth)

<Note>
These are sensible defaults for most teams. Only tune further if you observe concrete bottlenecks.
</Note>

## Optional: Add State Encryption

For production systems, enable OpenTofu state encryption:

```hcl
terraform {
  backend "oras" {
    repository   = "ghcr.io/myorg/infra-state"
    compression  = "gzip"
    lock_ttl     = 300
    max_versions = 10
  }

  encryption {
    key_provider "pbkdf2" "main" {
      passphrase = var.state_passphrase
    }
    method "aes_gcm" "main" {
      key_provider = key_provider.pbkdf2.main
    }
    state { method = method.aes_gcm.main }
    plan  { method = method.aes_gcm.main }
  }
}
```

<Warning>
For real production systems, prefer a KMS-backed key provider over PBKDF2 passphrases.
</Warning>

## Verify State Storage

Check that your state was pushed to GHCR:

1. Navigate to your GitHub organization or user profile
2. Go to **Packages**
3. Find the `infra-state` package
4. You should see tags like:
   - `state-default` (current state for default workspace)
   - `locked-default` (active lock, if any)
   - `state-default-v1`, `state-default-v2` (historical versions)

## Common Issues

<AccordionGroup>
  <Accordion title="Authentication failed">
    **Problem:** `unauthorized` or `denied` errors
    
    **Solution:** Re-run `docker login`. On GHCR, ensure your token has `read:packages` and `write:packages` scopes.
  </Accordion>
  
  <Accordion title="Lock appears stuck">
    **Problem:** Lock is held after a crashed job
    
    **Solution:** Set `lock_ttl = 300` in your backend config to auto-clear stale locks on next acquisition.
  </Accordion>
  
  <Accordion title="Cannot delete versions on GHCR">
    **Problem:** Version deletion fails
    
    **Solution:** Ghoten falls back to GitHub Packages API. Ensure your token has package delete permission.
  </Accordion>
</AccordionGroup>

<Note>
For deep diagnostics, always run with `TF_LOG=DEBUG ./ghoten plan`.
</Note>
